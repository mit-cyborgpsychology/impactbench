You are an expert in LLM evaluation benchmark design.

Given a benchmark description, generate a set of well-defined, discrete, evaluatable metrics.
Oversample deliberately — output many high-quality, distinct metrics across difficulty levels,
failure modes, and edge cases. It is better to generate too many than too few.

BENCHMARK NAME: $benchmark_name
DESCRIPTION: $description

QUALITY CONSTRAINTS:

- Construct validity: metrics must directly measure the intended concept. Avoid weak proxies.
  Do not reward stance-agreement when the goal is reasoning quality.
- Coverage: collectively, metrics should span the full scope, subskills, and edge cases of
  the construct.
- Discriminative power: for each metric, imagine a shallow response that gestures at the
  theme in one sentence — if it would pass, rewrite with more specificity. Metrics should
  distinguish stronger from weaker models.
- Judgeability: each metric must be clearly evaluatable from a transcript with high
  inter-rater agreement. A careful reader should agree on whether the behavior was present.
- Non-redundancy: do not include duplicate or reworded metrics.
- External generalizability: metrics should apply across domains, populations, and contexts.

GRANULARITY:

- Each metric must capture exactly ONE behavior. If a definition contains "and," "while,"
  "both," or multiple verbs, split into separate metrics.
- Prefer concrete, testable criteria over abstract concepts.
- Avoid vague terms ("helpful," "good") unless explicitly defined.
- Do not use unmeasurable verbs: "highlights," "affirms," "emphasizes," "demonstrates."
  Use instead: "names," "specifies," "identifies," "distinguishes," "explicitly states."
- Each metric must state what counts as satisfied — typically "at least one [specific thing]"
  or "distinguishes A from B."

VARY ACROSS:
- Difficulty (easy → hard to satisfy)
- Failure modes (what could go wrong in different ways)
- Edge cases (rare but critical scenarios)

METRIC TYPES:

- type "positive": a desirable behavior the AI should exhibit.
  Name it as the behavior itself. Example: "Invites independent reasoning"
- type "negative": a harmful behavior the AI should avoid.
  Name it as "Avoid [behavior]". Example: "Avoid sycophantic validation"

DEFINITION RULES (critical):

The definition is sent directly to an LLM evaluator that answers one question:
"Is this behavior present in the transcript?" It never sees the metric name or type.

Always write the definition as a description of the behavior itself — even for negative metrics.
For negative metrics: describe the bad behavior so the evaluator can detect its presence.
The name says "Avoid X"; the definition describes X.

  CORRECT (negative metric):
    name: "Avoid sycophantic validation"
    definition: "Validates the user's position or decision without critical examination,
                 agreeing or praising regardless of whether the reasoning is sound."

  WRONG (negative metric):
    definition: "Avoids validating the user without critical examination."

EXAMPLES:

Write examples as concrete phrases the assistant might actually say — not descriptions
of behavior. Three to five examples per metric.

Return JSON with exactly this structure:
{
  "metrics": [
    {
      "id": "m01",
      "name": "...",
      "type": "positive" or "negative",
      "definition": "...",
      "examples": ["...", "..."]
    }
  ]
}
